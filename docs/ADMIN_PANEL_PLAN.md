# Админ-панель платформы Manexa — концепция и план

> Идеи по панели администрирования платформы: заведение компаний, управление подписками, поддержка и полный контроль над инстансом.
>
> Связанные документы: `ПРОДАКШЕН_ИДЕИ.md` (подписки, billing), `docs/ROADMAP_AND_ISSUES.md` (P3 — коммерческий SaaS).
>
> *Составлено: Июнь 2026*

---

## 1. Зачем и для кого

Сейчас Manexa — один инстанс, компании создаются через публичную регистрацию (`/auth/signup` → новая `Company` + `OWNER`). Для коммерческой модели нужен контур **администратора платформы**, который:

1. **Заводит компании** вручную (продажа → онбординг → создание аккаунта директора).
2. **Управляет жизненным циклом** компании: триал → оплата → блокировка → архив.
3. **Видит здоровье системы**: использование ресурсов, ошибки, очереди, диск MinIO.
4. **Помогает клиентам**: сброс пароля директора, вход «от имени», диагностика данных.

Два уровня администрирования, не смешивать:

| Уровень | Кто | Что видит |
|---------|-----|-----------|
| **Платформа** (`/platform/*`) | PLATFORM_ADMIN / PLATFORM_MANAGER | Все компании, подписки, системные метрики. **Не видит** содержимое проектов/документов без impersonation |
| **Компания** (`/settings`, `/users`) | OWNER / ADMIN | Только своя компания — уже реализовано |

## 2. Роли и доступ

### 2.1. Новые роли

Расширить `enum UserRole` (сейчас: OWNER, ADMIN, MANAGER, USER):

```prisma
enum UserRole {
  PLATFORM_ADMIN    // полный доступ к платформе, управление менеджерами
  PLATFORM_MANAGER  // заведение компаний, поддержка, без системных настроек
  OWNER
  ADMIN
  MANAGER
  USER
}
```

Особенности платформенных ролей:

- `companyId = null` — не принадлежат ни одной компании.
- Не видны в списках `/users` компаний, не участвуют в проектах/задачах.
- Middleware: запрос к `/platform/*` и `/api/platform/*` — только для платформенных ролей; обычным пользователям — 404 (не 403, чтобы не раскрывать существование панели).
- Платформенные роли **не могут** обращаться к обычным API компаний (кроме режима impersonation) — изоляция в обе стороны.

### 2.2. Защита входа

- Отдельная страница входа `/platform/signin` (можно та же форма, но редирект по роли).
- **2FA обязательна** для платформенных ролей (TOTP, `speakeasy`) — внедрить раньше, чем 2FA для клиентов.
- IP-allowlist (опционально): список разрешённых IP в env (`PLATFORM_ALLOWED_IPS`), проверка в middleware.
- Сессия короче обычной (например, 4 часа вместо 30 дней).

## 3. Разделы панели `/platform/*`

### 3.1. Дашборд (`/platform`)

Сводка по платформе:

- Компаний всего / активных / в триале / заблокированных.
- Пользователей всего / активных за 7 дней (по `Session` или новой таблице активности).
- Новые компании и платежи за месяц (график).
- Системные индикаторы: статус PostgreSQL, Redis, MinIO, Gotenberg, размер очереди BullMQ, свободное место на диске (часть уже есть в `/monitoring` — переиспользовать `src/app/api/monitoring`).

### 3.2. Компании (`/platform/companies`)

Главный рабочий экран менеджера:

**Список:** название, ИНН, тариф, статус подписки, пользователей/лимит, проектов, дата создания, последняя активность. Фильтры по статусу и тарифу, поиск по названию/ИНН.

**Создание компании (мастер, 2 шага):**
1. Реквизиты компании — поля модели `Company` уже покрывают всё (ИНН/КПП/ОГРН, банк, директор). Подтянуть по ИНН через DaData (интеграция уже есть).
2. Аккаунт директора — имя, email, временный пароль (генерируется), флаг `mustChangePassword`. Отправка приглашения на email (когда появится email-сервис) или выдача пароля менеджером вручную.

**Карточка компании (`/platform/companies/[id]`):**
- Реквизиты + редактирование.
- Подписка: тариф, период, история платежей, кнопки «Продлить», «Сменить тариф», «Заблокировать/Разблокировать» (ручной режим до интеграции платёжки).
- Пользователи: список с ролями, кнопка «Сбросить пароль» (генерирует временный + `mustChangePassword`), деактивация.
- Использование: проектов, документов, объём файлов в MinIO (по префиксу bucket), сообщений чата.
- Опасная зона: архивирование компании (`isActive=false` — поле уже есть), полное удаление с экспортом данных.

### 3.3. Подписки и тарифы (`/platform/billing`)

Модели `Subscription`/`Payment` — взять из `ПРОДАКШЕН_ИДЕИ.md` §2 (уже спроектированы).

- Справочник тарифов: лимиты (пользователи, проекты, ГБ хранилища), цена. Хранить в БД (`Plan`), а не в коде — чтобы менять без деплоя.
- Список подписок с просрочками — главный экран «кого дёрнуть по оплате».
- Ручная фиксация оплаты (банковский перевод по счёту) — пока без онлайн-платёжки: менеджер отмечает платёж, период продлевается.
- Лимиты тарифа применять мягко: при превышении — баннер «обратитесь к менеджеру», жёсткая блокировка только создания новых сущностей.

### 3.4. Пользователи платформы (`/platform/users`)

- Сквозной поиск пользователя по email (в какой компании, какая роль, активен ли) — первое, что нужно поддержке.
- Управление платформенными менеджерами (только PLATFORM_ADMIN): создание, 2FA-reset, отзыв доступа.

### 3.5. Вход от имени (impersonation)

Самый ценный инструмент поддержки и самый опасный:

- Кнопка «Войти как директор» в карточке компании.
- Реализация: отдельный короткоживущий токен (15–30 мин), сессия помечена `impersonatedBy: <platformUserId>`.
- В UI клиента — несмываемая плашка «Вы вошли как X (поддержка). Выйти».
- **Каждое действие** в режиме impersonation пишется в audit log.
- Только PLATFORM_ADMIN, либо PLATFORM_MANAGER с подтверждением (запрос/одобрение).

### 3.6. Аудит (`/platform/audit`)

Модель:

```prisma
model PlatformAuditLog {
  id        String   @id @default(cuid())
  actorId   String   // платформенный пользователь
  action    String   // COMPANY_CREATE, SUBSCRIPTION_EXTEND, IMPERSONATE_START, PASSWORD_RESET...
  targetType String? // Company, User, Subscription
  targetId  String?
  metadata  Json?
  ip        String?
  createdAt DateTime @default(now())

  @@index([actorId])
  @@index([targetType, targetId])
  @@index([createdAt])
}
```

Логировать **все** действия платформенных ролей. Просмотр с фильтрами, без возможности удаления через UI.

### 3.7. Система (`/platform/system`) — только PLATFORM_ADMIN

- Здоровье сервисов (расширение текущего `/monitoring`): БД, Redis, MinIO, Gotenberg, PM2-процессы, очередь экспорта.
- Бэкапы: статус последнего `pg_dump`, кнопка «создать бэкап сейчас», список бэкапов (см. `ПРОДАКШЕН_ИДЕИ.md` §5.1).
- Глобальные флаги: `ALLOW_PUBLIC_REGISTRATION` (перенести из env в БД-настройку), режим обслуживания (баннер всем компаниям), feature flags по компаниям (например, включить КС-2/КС-3 пилотной компании).
- Журнал ошибок приложения (интеграция с `src/lib/alerts.ts` / Sentry).

## 4. Изменения в существующем коде

| Что | Где | Заметка |
|-----|-----|---------|
| Роли PLATFORM_* | `prisma/schema.prisma`, `src/lib/permissions.ts` | Добавить в enum + права `canManagePlatform`, `canImpersonate` |
| Отключение публичной регистрации | `/auth/signup`, `src/app/api/auth/register` | Флаг `ALLOW_PUBLIC_REGISTRATION` (уже упоминается в roadmap §3.5) |
| `mustChangePassword` | `User` + страница принудительной смены | Нужен и для сброса пароля поддержкой |
| Middleware изоляции | `src/middleware.ts` | `/platform/*` → только платформенные роли; платформенные роли → запрет обычных API |
| Блокировка компании | `checkPermission` / middleware | Если `company.isActive=false` или подписка SUSPENDED → редирект на `/suspended` |
| Деактивация пользователя | уже сделано (июнь 2026) | Проверка `isActive` при входе реализована — переиспользовать для блокировок |

## 5. Этапы внедрения

### Этап 1 — MVP панели (1–1.5 недели)
1. Роли `PLATFORM_ADMIN` / `PLATFORM_MANAGER`, seed первого админа, middleware-изоляция.
2. `/platform/companies`: список, создание компании + директора (`mustChangePassword`), карточка, архивирование.
3. Отключение публичной регистрации (флаг).
4. `PlatformAuditLog` — логирование с первого дня.
5. Сквозной поиск пользователя, сброс пароля.

### Этап 2 — подписки без платёжки (1 неделя)
1. Модели `Plan` / `Subscription` / `Payment`, триал 14 дней при создании компании.
2. Ручная фиксация оплат, продление, блокировка `/suspended`.
3. Cron проверки статусов (расширить существующий воркер).
4. Дашборд `/platform` с метриками.

### Этап 3 — поддержка и система (1 неделя)
1. Impersonation с плашкой и аудитом.
2. 2FA для платформенных ролей.
3. `/platform/system`: health, бэкапы, режим обслуживания, feature flags.

### Этап 4 — автоматизация (по мере роста)
1. Онлайн-оплата (ЮKassa/CloudPayments — см. `ПРОДАКШЕН_ИДЕИ.md` §2.3).
2. Email-уведомления (приглашение, просрочка, блокировка).
3. Аналитика использования по компаниям, экспорт.
4. Self-service регистрация с модерацией (заявка → одобрение менеджером).

## 6. Открытые вопросы

1. **Один инстанс или несколько?** Пока один сервер (`manexa-linux`) на LAN. Для внешних клиентов нужен VPS + HTTPS (Caddy) — панель проектировать сразу под один общий инстанс с мультитенантной изоляцией.
2. **Где хранить тарифы** — в БД (рекомендуется) или конфиге.
3. **Импersonation для PLATFORM_MANAGER** — разрешить всем менеджерам или только по одобрению админа.
4. **Удаление компании** — физическое удаление или только архив + экспорт дампа (рекомендуется архив; физическое удаление — отдельной процедурой по запросу).
5. **Email-сервис** — без него приглашения директорам выдаются вручную (пароль в руки). Достаточно ли для старта?
